# What are Projects?

## The One-Sentence Answer

A project is a workspace that maps to a directory on your filesystem, containing all the skills, agents, tools, and configuration for one codebase or initiative.

## The Analogy: Folders on Your Desk

If you're working on three products, you probably have three folders — each with its own documents, checklists, and notes. You don't mix documents between folders.

A project in Orkestr is that folder. Everything inside it — skills, agent configurations, MCP connections, workflow definitions — is scoped to that project.

## What a Project Contains

```
Project: "Backend API"
├── Skills (15 skills)
│   ├── coding-standards.md
│   ├── api-design-guide.md
│   ├── security-checklist.md
│   └── ...
├── Agents (4 enabled)
│   ├── Code Review Agent (custom instructions + 5 skills)
│   ├── Security Agent (custom instructions + 3 skills)
│   ├── QA Agent (custom instructions + 4 skills)
│   └── Architect Agent (custom instructions + 3 skills)
├── MCP Servers (2 connected)
│   ├── Filesystem (local project files)
│   └── GitHub (PR management)
├── Workflows (2 defined)
│   ├── PR Review Pipeline
│   └── Nightly Security Scan
├── Provider Sync (3 enabled)
│   ├── Claude ✓
│   ├── Cursor ✓
│   └── Copilot ✓
└── Settings
    ├── Git auto-commit: enabled
    ├── Default model: claude-sonnet-4-6
    └── Guardrail profile: moderate
```

## Project ↔ Filesystem

Every project maps to a real directory:

```
/home/user/projects/backend-api/
├── .orkestr/                    ← Orkestr's directory
│   └── skills/                  ← Skill files live here
│       ├── coding-standards.md
│       ├── api-design-guide.md
│       └── security-checklist.md
├── .claude/                     ← Generated by provider sync
│   └── CLAUDE.md
├── .cursor/                     ← Generated by provider sync
│   └── rules/
│       ├── coding-standards.mdc
│       └── api-design-guide.mdc
├── src/                         ← Your actual code
│   ├── auth/
│   ├── api/
│   └── models/
└── ...
```

The `.orkestr/` directory is the single source of truth. The `.claude/`, `.cursor/`, etc. directories are **generated outputs** — you never edit them directly.

## Creating a Project

Projects are created in the Settings panel or Filament admin:

1. **Name** — "Backend API"
2. **Path** — `/home/user/projects/backend-api`
3. **Providers** — Check which AI tools to sync to
4. **Settings** — Git auto-commit, default model, guardrail profile

## Scanning Existing Skills

If your project already has `.orkestr/skills/*.md` files (maybe from another team member or a previous installation), click **Scan**. Orkestr reads every file, parses the YAML frontmatter, and imports the skills into the database.

## Per-Project Configuration

Each project has its own:

| Setting | What It Controls |
|---|---|
| **Enabled agents** | Which of the 9 pre-built agents are active |
| **Custom instructions** | Project-specific additions to each agent |
| **Skill assignments** | Which skills are assigned to which agents |
| **Template variable values** | Per-project overrides for `{{variable}}` placeholders |
| **Provider sync targets** | Which AI coding tools to generate config for |
| **MCP server bindings** | Which tool servers are available |
| **Guardrail profile** | Safety policy level |
| **Git auto-commit** | Whether to commit on every skill save |

## Multi-Project Workflows

While most work is scoped to a single project, some features work across projects:

- **Cross-project search** — Find skills across all projects
- **Skill duplication** — Clone a skill from one project to another
- **Library** — Shared skill library available to all projects
- **Bundles** — Export skills from one project, import into another

## Key Takeaway

Projects are the organizational unit of Orkestr. Each project maps to a codebase, contains its own skills, agents, and configuration, and keeps everything isolated. The `.orkestr/` directory is the single source of truth that lives in your git repo.
